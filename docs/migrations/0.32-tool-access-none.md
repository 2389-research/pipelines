# Migration: `tool_access: none` (dippin v0.32.0)

Short note for anyone using these pipelines as templates.

## What changed

dippin **v0.32.0** added a language-level `tool_access: none` field on `agent`
nodes. When set, the engine strips the model's entire tool catalog
(`Read`, `Write`, `Edit`, `ApplyPatch`, `Glob`, `GrepSearch`, `Bash`) at the
language level — the model is invoked with no tools at all. Before v0.32.0 the
native backend unconditionally shipped the full catalog to every agent and there
was no structural way to suppress it; the only available bound was prose in the
`prompt:` body, which the model can ignore. This closed
[dippin-lang#41](https://github.com/2389-research/dippin-lang/issues/41).

In this repo the field is used on acknowledge-only `Start`/`Exit` agents and on
LLM nodes that should reason over context but never touch the filesystem — for
example the issue-selection and squad-review agents in
[`dev_loop/dev_loop.dip`](../../dev_loop/dev_loop.dip)
(`tool_access: none` paired with `system_prompt_file:`/`prompt_file:`).

## Tool access vs. policy: which prose to drop, which to keep

`tool_access: none` replaces a specific kind of prose — instructions that exist
**only** to deny tool use. It does **not** replace task policy.

- **Drop / demote to belt-and-suspenders:** lines whose sole job is to deny tool
  use, e.g. `HARD CONSTRAINT: do not write files`, `do not run any tools`,
  `do not edit anything`. With `tool_access: none` the model has no tools, so the
  constraint is now enforced structurally. You can delete these or keep one line
  as documentation, but they are no longer the guard.
- **Keep verbatim:** every task-scoping / behavioral policy clause —
  `Your ONLY job is X`, `Do NOT fabricate evidence`, `respond with exactly one
  of …`, output-format contracts, refusal rules. These bound *what the model
  says*, not *what it can do*, and `tool_access: none` does nothing for them.

The two concerns are independent: stripping tools does not sanitize the prompt
or the incoming `${ctx.*}` context, and policy prose does not strip tools.

## How to adopt in your own `.dip` files

Add the field to any agent that should run with no filesystem access:

```dippin
agent Exit
  label: Exit
  tool_access: none
  max_turns: 1
```

Caveats:

- **All-or-nothing on the native backend.** `tool_access: none` removes the
  *entire* catalog, including `Read`. For a node that legitimately needs to read
  a file via its native `Read` tool, do **not** apply it — see the read-bounded
  waiver in [`../agent-node-safety.md`](../agent-node-safety.md).
- **Pair it with `max_turns: 1`** on terminal/acknowledge nodes to also bound
  iterative drift (the two address different failure modes).

## Compatibility

tracker **< v0.31.0** silently ignores the `tool_access` field — the node runs
with the full catalog as before, with no error. Pin tracker **≥ v0.31.0** (the
release that recognizes the field) so the bound actually takes effect. This repo
now pins higher; see [`0.32-to-current.md`](./0.32-to-current.md) for the current
toolchain.
